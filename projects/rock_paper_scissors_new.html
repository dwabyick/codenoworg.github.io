Rock, Paper, Scissors

In this project, students will build a procedural version of Rock, Paper, Scissors.

Learning Goals:

  • Understand program control flow and the process of writing pseudocode
  • Practice the fundamentals of Ruby including using variables (string), data structures (arrays), and control structures (conditionals: if/elsif/else), and understanding Boolean logic.

The design decisions in the following code are intentional to help absolute beginners quickly learn the basics without overwhelming them with intermediate concepts.

This tutorial is open source. If you notice errors, typos, or have questions/suggestions, please submit them to the project on Github.

Step 2 – Setting the computer and player moves

Review with students the concept of variables and data types (strings). See here and here for key concepts you want to touch on.

  1. Go over the purpose for which we would want to store information in our game, the scenarios during which we would want to manipulate the information (if any), the place in the program where we would want to use the information stored, etc.
  2. Ask the students to define variables to store the “computer move” and the “player move.”
    • Tell students to start off by setting both the computer and player moves to a fixed value.
  3. Have students run their programs to ensure that their code is in a working state.

The conditions in Rock, Paper, Scissors are slightly more complicated than the students covered in Hi-Lo.

Step 5: Adding randomness

We are going to use the sample method on the Array class in order to add randomness to the computer_move variable.

First, we’ll need to take a small detour and cover the concept of an array. See here for key concepts you want to touch on.

In order to randomly generate a computer move, we’ll have to first define an array of possible moves. Challenge the students to come up with an array and sample the array to pick a computer move.

computer_move is no longer fixed. Since the computer’s move is random, students should have difficulty proving all nine cases. Show them how they can debug their programs by printing out computer_move using puts.

Step 6 – Let's review

Now that you have completed the base project, do a quick review with your students of the concepts they used.

  • Ask students to tell you, without looking at their notes, what are the concepts they learned about in this project. The objective is for students to mention the building blocks introduced and/or reviewed in the project.
    • Writing pseudocode, understanding program control flow, using variables (string), data structures (arrays), and control structures (conditionals: if/elsif/else), and understanding Boolean logic.
  • If student do not mention some of the building blocks, ask them about it specifically
  • Finally, ask students if they would like you to review any of the building blocks or if they have any additional questions.

Step 7 – Add-on's

After completing the base project and doing a quick review of the building blocks introduced and/or reviewed, work with your students on adding additional features to the project. Work on add-on’s only if the time allocated to this project has not ran out.

Suggestions for add-on’s

  1. Add levels to the game, where each level has one more object and let students create their own rules. For example, in the easy level the player can choose rock, paper, scissors, or mirror; etc.
  2. Add version options and let the students create their own rules. For example, the player can choose to play the version where the objects are a sword, a ruby, and a net.
  3. Add a “quit” option, where the player is asked if he/she wants to play or quit.
  4. Invalid input: modify the program such that it does not accept invalid inputs
  5. Refactoring: There are some refactorings that can be done. Explain the concept of refactoring and why it is useful to clean up code. See here for key concepts you want to touch on. A refactor students can do is modifying the program to only use one if statement for the three tie scenarios. Challenge the students to write this refactor by themselves. As a hint tell students to think about to comparing the player’s move to the computer’s move.
  6. Let students come up with their own ideas on how to extend their programs - see what they come up with!
